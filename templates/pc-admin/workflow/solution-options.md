# PC Admin 实现方案选项

## 1. 背景

`workflow/requirements.md` 已确认。当前目标是为 PC Admin 模板选择一个 implementation-ready 前的技术路线，后续只能在用户明确选择方案后继续生成 `workflow/solution-selected.md` 和 implementation-ready 产物。

三个方案都遵守以下固定合同：

- Monorepo 保持 `frontend` + `backend`。
- 前端技术栈保持 Vue 3 + Vite 7 + TypeScript + Element Plus + Tailwind CSS 4 + Vue Router 4 + Pinia + Axios。
- 后端技术栈保持 Fastify 5 + TypeScript + MySQL2 + JWT + Winston。
- `SPECS/API.md` 是唯一前后端共享 API 契约。
- 第一阶段权限控制到菜单级。
- 不在 solution-options 阶段实现功能代码。

## 2. 依赖图

```text
数据库 schema 与 seed
  -> 字段映射与共享 API 契约
    -> 后端基础设施：Fastify / MySQL pool / JWT / logger / error handler
      -> 认证与当前用户 / 动态菜单
        -> 前端请求封装 / Pinia 用户态 / 动态路由
          -> 首页与后台管理模块
            -> 日志、消息、附件、配置等扩展闭环
```

必须顺序推进：

- 数据库表和默认数据先于真实 API。
- 登录认证先于所有登录态模块。
- 菜单权限先于动态路由和侧边栏。
- 用户、角色、菜单是权限闭环核心，先于部门、岗位、字典、配置等周边维护。

可并行推进：

- 前端静态页面骨架和后端基础设施可并行，但必须先冻结 API 字段。
- 部门、岗位、字典、配置属于同类基础 CRUD，可在认证和通用 CRUD 模式稳定后并行。
- 消息、日志、附件可以在核心管理闭环可用后并行。

## 3. 方案 A：demo-vertical-slice

### 定位

优先跑通最小但真实的纵向闭环：登录、当前用户、动态菜单、首页、用户/角色/菜单基础维护。其余 13 个模块先保留 API/数据库合同，按优先级逐步补齐。

### 实施策略

第一阶段只实现能证明模板价值的核心链路：

1. 数据库 schema、seed、默认账号、默认角色、默认菜单。
2. 后端基础设施：Fastify app、MySQL pool、JWT、统一响应、错误处理、Winston。
3. 登录、刷新 Token、退出、当前用户、动态菜单。
4. 前端 Axios、Pinia 用户态、登录页、动态路由、基础 layout。
5. 首页 overview。
6. 用户、角色、菜单的基础查询和维护。

部门、岗位、字典、配置、消息、日志、附件、个人信息按后续任务追加，但预留菜单和 API 路径。

### 优点

- 见效最快，最适合先证明 generated project 可真实联调。
- 风险集中在认证、菜单、路由、字段映射这些最高价值路径。
- 单次实现面较小，更容易用 TDD 和手工验证控制质量。

### 缺点

- 13 个模块不会一次性全量可用。
- 需求文档和实际功能会短期存在“合同已全量定义、实现分阶段补齐”的落差。
- 日志、附件、消息、系统配置等模块的集成价值要到后续阶段才显现。

### 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 后续模块接入时发现 API 字段不够 | 中 | 先以 `SPECS/API.md` 冻结核心字段，新增字段必须同步契约 |
| 菜单权限和动态路由调试复杂 | 高 | 把登录、`/auth/me`、`/get-async-routes` 作为第一个验收 checkpoint |
| Demo 显得模块不完整 | 中 | 首页和菜单显示全部模块入口，但未实现模块明确返回建设中或后续任务 |

### 验证点

- 超级管理员和普通用户可登录。
- 两类账号菜单不同。
- 首页来自真实接口。
- 用户、角色、菜单能完成基础维护。
- `pnpm typecheck`、`pnpm build` 通过。

### 适用场景

选择这个方案，如果当前最重要的是尽快进入可运行 demo，并接受分阶段补齐剩余模块。

## 4. 方案 B：backend-contract-first

### 定位

先把后端 API、数据库、seed 和接口行为做成完整合同层，前端只优先接入认证、动态菜单、首页和少量管理页。此方案把后端作为模板可信度核心。

### 实施策略

第一阶段重点铺后端：

1. 完整 MySQL schema 和 seed。
2. 后端基础设施、统一响应、错误码、鉴权、日志中间件。
3. 按 `SPECS/API.md` 建立 13 个模块的 route 骨架。
4. 每个模块至少实现分页查询、详情、基础维护或明确的 demo 级行为。
5. 前端优先接入登录、动态菜单、首页、用户/角色/菜单。
6. 其他前端模块可先使用通用列表/表单壳或延后。

### 优点

- 前后端共享 API 契约最早被验证。
- 数据库设计和字段映射风险暴露更早。
- 后续前端模块开发会更稳定，不需要频繁等后端补接口。

### 缺点

- 初期用户可见成果慢于方案 A。
- 后端一次性覆盖 13 个模块，任务拆分和质量控制压力更大。
- 若没有足够测试，容易出现“接口很多但深度不够”的问题。

### 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 后端任务面过大 | 高 | 每个模块限制 demo 级 CRUD，复杂能力只留扩展点 |
| 前端接入滞后 | 中 | 保证登录、菜单、首页三个可视入口优先完成 |
| 数据库 schema 修改牵连大 | 高 | 在 implementation-ready 前冻结 schema 草案，后续变更必须记录决策 |

### 验证点

- 13 个模块的 API 路由均存在。
- 核心模块返回真实 MySQL 数据。
- 登录态、权限菜单、统一错误响应可用。
- API 字段与 `SPECS/API.md` 对齐。
- `pnpm --filter fullstack-admin-backend typecheck` 和后端 build 通过。

### 适用场景

选择这个方案，如果当前最重要的是把后端模板和 API 契约做扎实，让前端后续按稳定合同推进。

## 5. 方案 C：full-module-foundation

### 定位

一次性建立 13 个模块的前后端基础可用版本。每个模块都至少有菜单入口、列表页、查询、基础维护接口和数据库落点，形成完整后台模板骨架。

### 实施策略

第一阶段铺完整模块框架：

1. 完整数据库 schema 和 seed。
2. 后端通用 CRUD helper、分页 helper、字段映射、错误处理、操作日志钩子。
3. 13 个模块 route/service 基础实现。
4. 前端通用列表页、表单弹窗、状态切换、字典选项、分页查询模式。
5. 所有模块菜单入口和基础页面。
6. 登录、动态路由、首页、权限菜单作为主验收闭环。

### 优点

- 模板完整度最高，最贴近 13 个模块需求文档。
- 后续业务扩展有统一页面和接口范式。
- 可以较早发现模块之间的字段、字典、日志、配置联动问题。

### 缺点

- 初始实现规模最大，最容易超出 demo 级边界。
- 单次变更文件多，回归测试和人工验证成本高。
- 如果抽象过早，可能产生不必要的通用 CRUD 层复杂度。

### 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 任务过大导致实现质量不稳定 | 高 | implementation-ready 阶段必须拆成小任务，每任务不超过约 5 个文件 |
| 通用 CRUD 抽象过重 | 中 | 只抽取分页、响应、字段映射等确定重复点 |
| 前端页面重复度高 | 中 | 使用统一列表/表单模式，但避免生成难维护的大型配置 DSL |

### 验证点

- 13 个模块都有菜单入口。
- 13 个模块至少具备查询页或基础维护页。
- 后端 13 个模块至少有 route/service 骨架和真实数据库访问。
- 登录、权限、首页、日志、消息、配置、附件至少能演示核心行为。
- 根目录 `pnpm typecheck`、`pnpm build` 通过。

### 适用场景

选择这个方案，如果当前最重要的是模板完整度，并且可以接受更长实现周期和更严格任务拆解。

## 6. 推荐

推荐选择 `demo-vertical-slice`。

理由：

- 当前需求范围包含 13 个模块，但目标明确是 demo 级真实联调；最重要的成功信号是 generated project 能跑出登录、权限菜单、首页和核心管理闭环。
- 该方案最符合依赖图：先做数据库 seed、认证、菜单、动态路由，再向周边模块扩展。
- 它能最快暴露前后端字段映射、Token 刷新、菜单权限、Vite proxy 和 MySQL 联调这些系统性风险。
- 它避免在第一轮把 13 个模块全部铺开，降低任务过大和抽象过早的概率。

不建议第一轮选择 `full-module-foundation`，除非你愿意把实现周期显著拉长，并接受更强的任务治理和回归验证成本。

## 7. 冻结点

在用户明确选择方案前：

- 不创建 `workflow/solution-selected.md`。
- 不创建 `workflow/implementation-ready.md`。
- 不拆具体实现任务。
- 不修改前端或后端功能代码。

用户选择后，下一步产物应是：

- `workflow/solution-selected.md`
- `memory/decisions.md` 中记录选择理由
- implementation-ready 前的具体任务拆解
